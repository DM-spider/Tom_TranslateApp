# 版本更新 —— 服务器问题排查总结

> **背景**：完成登录功能 + 付费计划后，将新代码部署到服务器时遇到的问题记录。当前部署方式为 Docker Compose 全容器化 + 宿主机 Nginx 反向代理。

---

## 部署架构

```
浏览器 → Nginx (宿主机, 443/80)
              ├── tomtranslate.com       → web 容器 (:3000)
              └── api.tomtranslate.com   → api 容器 (:8000) → redis / db / libretranslate
```

- 一张多域名 SSL 证书覆盖 `tomtranslate.com`、`www.tomtranslate.com`、`api.tomtranslate.com`
- 所有容器端口只绑定 `127.0.0.1`，外部流量必须经过 Nginx
- `NEXT_PUBLIC_API_URL=https://api.tomtranslate.com`

## 问题详解

### 问题 1：注册接口 500 —— passlib 与 bcrypt 不兼容

**现象**：`POST /api/v1/auth/register` 返回 `Internal Server Error`。

**日志**：

```
(trapped) error reading bcrypt version
AttributeError: module 'bcrypt' has no attribute '__about__'
```

**根因**：`passlib` 已停止维护，与 `bcrypt >= 4.0.0` 不兼容（新版 bcrypt 移除了 `__about__` 模块）。

**解决方案**：移除 passlib 依赖，直接使用 bcrypt 库进行密码哈希：

```python
import bcrypt

def hash_password(password: str) -> str:
    return bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

def verify_password(plain: str, hashed: str) -> bool:
    return bcrypt.checkpw(plain.encode(), hashed.encode())
```

**教训**：升级依赖前需检查间接依赖的兼容性。优先使用活跃维护的库。

---

### 问题 2：Docker build 缓存导致修复未生效

**现象**：修改代码后 `docker compose up -d --build api`，所有步骤显示 `CACHED`，容器仍报同样的错误。

**根因**：
1. 本地代码修改后没有 `git push`，服务器 `git pull` 显示 `Already up to date`
2. Docker 使用了旧的构建缓存层

**解决方案**：

```bash
# 1. 本地推送代码
git add -A && git commit -m "fix: ..." && git push

# 2. 服务器拉取后用 --no-cache 重建
git pull
docker compose -f deploy/docker-compose.prod.yml build --no-cache api
docker compose -f deploy/docker-compose.prod.yml up -d api
```

**注意**：`--no-cache` 是 `build` 子命令的参数，不是 `up` 的参数。正确用法是 `build --no-cache`。

**教训**：修改代码后部署必须确保：(1) 代码已 push 到远端 (2) 服务器已 pull (3) 使用 `--no-cache` 重建镜像。

### 问题 3：SSL 证书不匹配 api.tomtranslate.com

**现象**：浏览器报 `ERR_CERT_COMMON_NAME_INVALID`，所有发往 `api.tomtranslate.com` 的请求失败。

**排查**：

```bash
openssl s_client -connect api.tomtranslate.com:443 -servername api.tomtranslate.com \
  | openssl x509 -noout -text | grep -A1 "Subject Alternative Name"
# → DNS:tomtranslate.com, DNS:www.tomtranslate.com  （不包含 api.tomtranslate.com）
```

**根因**：申请 Certbot 证书时只覆盖了主域名和 www，漏掉了 `api.tomtranslate.com`。

**解决方案**：重新申请证书，包含所有三个域名：

```bash
certbot --nginx -d tomtranslate.com -d www.tomtranslate.com -d api.tomtranslate.com
```

Nginx 配置中三个域名共用这张证书（指向同一路径 `/etc/letsencrypt/live/tomtranslate.com/`）。

**教训**：申请 SSL 证书时，必须一次性包含所有需要的域名（主域名 + www + api 子域名）。

### 问题 4：HTTPException 被 except Exception 吞掉，403 变 502

**现象**：未登录使用 AI 引擎翻译，应返回 403 提示"需要登录"，但前端收到 502。

**日志**：

```
ERROR:root:翻译引擎调用失败
fastapi.exceptions.HTTPException: 403: 使用 AI 翻译需要登录，免费机器翻译无需登录
INFO:  "POST /api/v1/translate HTTP/1.0" 502 Bad Gateway
```

**根因**：`translate.py` 中 `except Exception as e:` 把 FastAPI 的 `HTTPException(403)` 也捕获了，统一以 502 重新抛出。`HTTPException` 继承自 `Exception`。

**解决方案**：在 `except Exception` 之前加上 `except HTTPException: raise`：

```python
except RedisConnectionError:
    raise HTTPException(status_code=503, detail="缓存服务暂时不可用，请稍后重试")
except HTTPException:        # ← 让 403/429 等正常返回给前端
    raise
except Exception as e:
    ...
    raise HTTPException(status_code=502, detail=safe_msg)
```

**教训**：在 FastAPI 中使用宽泛异常捕获时，必须在前面显式 `except HTTPException: raise`，否则业务状态码会被吞掉。

### 问题 5：机器翻译 502 —— LibreTranslate 服务未部署

**现象**：选择"机器翻译（免费）"时返回 502。

**根因**：`docker-compose.prod.yml` 中配置了 `LIBRE_TRANSLATE_URL=http://libretranslate:5000`，但没有在 compose 中定义 LibreTranslate 服务，api 容器连接该地址失败。

**解决方案**：在 `docker-compose.prod.yml` 中添加 LibreTranslate 服务：

```yaml
libretranslate:
  image: libretranslate/libretranslate:latest
  restart: always
  environment:
    - LT_LOAD_ONLY=zh,en,ja,ko,fr,de,es,ru
  volumes:
    - lt_data:/home/libretranslate/.local
  networks:
    - app-network
```

**注意**：LibreTranslate 首次启动需下载语言模型（约 1-2GB），可能耗时数分钟。`LT_LOAD_ONLY` 限制只加载必要语言以节省资源。

**教训**：在 compose `environment` 中引用服务名作为 URL 时，该服务必须也在 compose 中定义。

## 版本更新标准流程

```bash
# 1. 拉取最新代码
cd /root/Tom_TranslateApp
git pull

# 2. 重建并启动
# 只改了后端：
docker compose -f deploy/docker-compose.prod.yml build --no-cache api
docker compose -f deploy/docker-compose.prod.yml up -d api

# 只改了前端（NEXT_PUBLIC_API_URL 是编译时写入的，必须重建）：
docker compose -f deploy/docker-compose.prod.yml build --no-cache web
docker compose -f deploy/docker-compose.prod.yml up -d web

# 全部都改了：
docker compose -f deploy/docker-compose.prod.yml build --no-cache
docker compose -f deploy/docker-compose.prod.yml up -d

# 3. 验证
docker compose -f deploy/docker-compose.prod.yml ps
curl http://localhost:8000/health
curl -I http://localhost:3000
curl -I https://tomtranslate.com
curl https://api.tomtranslate.com/health
```
