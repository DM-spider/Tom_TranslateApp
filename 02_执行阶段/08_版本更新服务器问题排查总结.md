## 问题详解与解决方案

### 问题 1：注册接口 500 —— passlib 与 bcrypt 版本不兼容

**现象**：

```bash
curl -X POST http://localhost:8000/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"123456"}'
# → Internal Server Error
```

**日志关键报错**：

```
(trapped) error reading bcrypt version
AttributeError: module 'bcrypt' has no attribute '__about__'
```

**根因**：`passlib` 库已很久未更新，与 `bcrypt >= 4.0.0` 不兼容。新版 bcrypt 移除了 `__about__` 模块，导致 passlib 调用失败。

**解决方案（二选一）**：

- **临时方案**：降级 bcrypt 版本

  ```bash
  cd /root/Tom_TranslateApp/packages/backend
  uv pip install "bcrypt<4.0"
  # 验证：bcrypt==3.2.2 安装成功
  ```

- **长期方案**：移除 passlib 依赖，直接使用 bcrypt 库进行密码哈希（已在本地代码中完成，需要部署到服务器）

**教训**：升级依赖前需要检测间接依赖的兼容性。

---

### 问题 2：网页 502 —— 前端服务未启动

**现象**：浏览器访问 `tomtranslate.com` 返回 502 Bad Gateway。

**排查过程**：

```bash
# 1. 确认后端正常
curl http://localhost:8000/health
# → {"status":"ok"}  ✅

# 2. 检查 nginx 配置，发现前端走 port 3000
cat /etc/nginx/conf.d/*.conf
# location / { proxy_pass http://127.0.0.1:3000; }

# 3. 检查前端进程
ps aux | grep node
# → 没有 node 进程  ❌

# 4. 查看 nginx 错误日志确认
tail -30 /var/log/nginx/error.log
# → connect() failed (111: Connection refused) ... upstream: "http://127.0.0.1:3000/"
```

**根因**：前端 Next.js 服务（port 3000）没有启动，Nginx 反向代理无法连接。

**解决方案**：

```bash
# 启动前端服务
cd /root/Tom_TranslateApp
pnpm run dev &
# 前端启动后：Next.js 15.5.12 - Local: http://localhost:3000 ✅
```

**教训**：服务器上的前后端服务需要分别确认是否在运行。可通过 `ps aux | grep node` 和 `ps aux | grep uvicorn` 快速排查。

### 问题 3：后端端口占用

**现象**：

```
ERROR: [Errno 98] Address already in use
```

**根因**：之前用 `nohup` 启动的 uvicorn 进程仍在运行，占据 8000 端口。

**解决方案**：

```bash
# 1. 找到占用端口的进程
ps aux | grep uvicorn

# 2. 终止旧进程
kill <旧PID>

# 3. 重新启动
cd /root/Tom_TranslateApp/packages/backend
nohup uv run uvicorn app.main:app --host 0.0.0.0 --port 8000 > /tmp/backend.log 2>&1 &
```

**教训**：重启服务前，先确认旧进程已退出。可以用 `lsof -i :8000` 或 `ps aux | grep uvicorn` 检查。

---

### 问题 4：Git pull 冲突

**现象**：

```
error: Your local changes to the following files would be overwritten by merge:
        packages/backend/alembic/env.py
Please commit your changes or stash them before you merge.
Aborting
```

**根因**：之前在服务器上直接修改过 `alembic/env.py`，导致拉取新代码时产生冲突。

**解决方案**：

```bash
# 方案 A：暂存本地修改再拉取
git stash
git pull
git stash pop  # 如需要恢复本地修改

# 方案 B：放弃本地修改直接拉取
git checkout -- packages/backend/alembic/env.py
git pull
```

**教训**：
- 尽量避免在服务器上直接修改代码，所有修改应通过 Git 从本地推送。
- 如果确实需要在服务器上临时修改，应第一时间 commit 或 stash，以避免后续 pull 冲突。

## 第二轮问题详解（04-07 Docker Compose 部署）

> 背景：将部署方式从 nohup 手动启动切换到 Docker Compose 全容器化（web + api + redis + db + libretranslate），暴露了多个本地与容器环境差异的问题。

### 问题 5：API 容器启动崩溃 —— PROJECT_ROOT 路径层级越界

**现象**：api 容器持续 `Restarting (1)`。

**日志关键报错**：

```
File "/app/app/config.py", line 22, in <module>
    PROJECT_ROOT = Path(__file__).resolve().parents[3]
IndexError: 3
```

**根因**：
- 本地路径 `packages/backend/app/config.py` → `parents[3]` = 项目根目录 ✅
- 容器内路径 `/app/app/config.py` → 只有 3 层（`/app/app` → `/app` → `/`），`parents[3]` 越界 ❌

**解决方案**：

```python
# 修改 packages/backend/app/config.py
_config_path = Path(__file__).resolve()
try:
    PROJECT_ROOT = _config_path.parents[3]
except IndexError:
    PROJECT_ROOT = _config_path.parent.parent
```

**教训**：硬编码的路径层级在不同运行环境（本地 vs Docker）下会不一致，应使用容错方式处理。

---

### 问题 6：Docker build 缓存导致修复未生效

**现象**：修改代码后执行 `docker compose up -d --build api`，所有步骤显示 `CACHED`，容器仍报同样的错误。

**根因**：
1. 本地代码修改后没有 `git push`，服务器 `git pull` 显示 `Already up to date`
2. Docker 使用了旧的构建缓存层

**解决方案**：

```bash
# 1. 本地 push 代码
git add -A && git commit -m "fix: ..." && git push

# 2. 服务器 pull 后用 --no-cache 重建
git pull
docker compose -f deploy/docker-compose.prod.yml build --no-cache api
docker compose -f deploy/docker-compose.prod.yml up -d api
```

**注意**：`--no-cache` 是 `build` 子命令的参数，不是 `up` 的参数。正确用法是 `build --no-cache`，而非 `up --no-cache`。

**教训**：Docker 修改代码后必须确保：(1) 代码已推送到服务器 (2) 使用 `--no-cache` 避免缓存。

---

### 问题 7 & 8：环境变量校验失败 —— localhost 在生产模式被拦截

**现象**：api 容器启动报 `DATABASE_URL 不能在 docker_host_nginx 模式下使用 localhost`，修复后又报 `LIBRE_TRANSLATE_URL` 同样的错误。

**根因**：`config.py` 中的 `validate_security_defaults` 校验器会在 `DEPLOYMENT_MODE != "local"` 时，拦截所有使用 `localhost/127.0.0.1/0.0.0.0` 的服务地址。compose 设了 `DEPLOYMENT_MODE=docker_host_nginx`，但只覆盖了 `REDIS_URL`，漏掉了 `DATABASE_URL` 和 `LIBRE_TRANSLATE_URL`。

**解决方案**：在 `docker-compose.prod.yml` 的 `api.environment` 中补齐所有服务地址：

```yaml
environment:
  - DEPLOYMENT_MODE=docker_host_nginx
  - REDIS_URL=redis://redis:6379/0
  - DATABASE_URL=postgresql+asyncpg://translate:translate_prod@db:5432/translate_db
  - LIBRE_TRANSLATE_URL=http://libretranslate:5000
```

**教训**：compose 覆盖环境变量时，必须对照 `config.py` 中所有 `localhost` 默认值逐一检查，确保无遗漏。

---

### 问题 9：SSL 证书不匹配 api.tomtranslate.com

**现象**：浏览器控制台报 `ERR_CERT_COMMON_NAME_INVALID`，所有发往 `api.tomtranslate.com` 的请求失败。

**排查**：

```bash
openssl s_client -connect api.tomtranslate.com:443 -servername api.tomtranslate.com \
  | openssl x509 -noout -text | grep -A1 "Subject Alternative Name"
# → DNS:tomtranslate.com, DNS:www.tomtranslate.com  (不包含 api.tomtranslate.com)
```

**根因**：Certbot 证书只覆盖主域名，不包含 `api` 子域名。

**解决方案**：不使用 `api.tomtranslate.com` 子域名，改为同域路径模式。Nginx 已配好 `location /api/` 转发到后端，让前端请求 `https://tomtranslate.com/api/v1/...` 即可。

```yaml
# docker-compose.prod.yml
web:
  build:
    args:
      NEXT_PUBLIC_API_URL: ${NEXT_PUBLIC_API_URL:-https://tomtranslate.com}
```

**备选方案**：如需使用子域名，需重新申请证书：
```bash
certbot --nginx -d tomtranslate.com -d www.tomtranslate.com -d api.tomtranslate.com
```

**教训**：部署前需确认 SSL 证书覆盖范围与前端 API 地址是否匹配。同域路径模式比子域名模式更简单，无需额外证书。

### 问题 10：HTTPException 被 except Exception 吞掉，403 变 502

**现象**：未登录使用 AI 引擎翻译，应返回 403 提示"需要登录"，但前端收到 502。

**日志关键信息**：

```
ERROR:root:翻译引擎调用失败
fastapi.exceptions.HTTPException: 403: 使用 AI 翻译需要登录，免费机器翻译无需登录
INFO:  "POST /api/v1/translate HTTP/1.0" 502 Bad Gateway
```

**根因**：`translate.py` 中 `except Exception as e:` 排在 `except HTTPException:` 之前，把 FastAPI 的 `HTTPException(403)` 也捕获了，然后统一以 502 重新抛出。

**解决方案**：在 `except Exception` 之前加上 `except HTTPException: raise`：

```python
except RedisConnectionError:
    raise HTTPException(status_code=503, detail="缓存服务暂时不可用，请稍后重试")
except HTTPException:        # ← 新增：让 403/429 等正常返回
    raise
except Exception as e:
    ...
    raise HTTPException(status_code=502, detail=safe_msg)
```

**教训**：`except Exception` 会捕获 `HTTPException`（因为 HTTPException 继承自 Exception）。在 FastAPI 中使用宽泛异常捕获时，必须在前面加 `except HTTPException: raise`。

---

### 问题 11：机器翻译 502 —— LibreTranslate 服务未部署

**现象**：选择"机器翻译（免费）"翻译时仍然 502。

**根因**：compose 配置了 `LIBRE_TRANSLATE_URL=http://libretranslate:5000`，但没有实际部署 LibreTranslate 容器，api 连接该地址失败。

**解决方案**：在 `docker-compose.prod.yml` 中添加 LibreTranslate 服务：

```yaml
libretranslate:
  image: libretranslate/libretranslate:latest
  restart: always
  environment:
    - LT_LOAD_ONLY=zh,en,ja,ko,fr,de,es,ru
  volumes:
    - lt_data:/home/libretranslate/.local
  networks:
    - app-network
```

**注意**：LibreTranslate 首次启动需要下载语言模型（约 1-2GB），可能需要几分钟到十几分钟。使用 `LT_LOAD_ONLY` 限制只加载必要语言以节省资源。

**教训**：在 compose 中引用服务名作为 URL 时，必须确保该服务也在 compose 中定义。

## 服务器版本更新标准流程（总结）

> 04-07 起已切换为 Docker Compose 全容器化部署，以下流程取代旧的 nohup 手动启动方式。

### 当前部署方式：Docker Compose

```bash
# ===== 1. 拉取最新代码 =====
cd /root/Tom_TranslateApp
git pull

# ===== 2. 重建并启动所有服务 =====
# 如果只改了后端代码：
docker compose -f deploy/docker-compose.prod.yml build --no-cache api

# 如果全部都改了：
docker compose -f deploy/docker-compose.prod.yml build --no-cache
docker compose -f deploy/docker-compose.prod.yml up -d

# ===== 3. 验证 =====
# 确认所有容器状态
docker compose -f deploy/docker-compose.prod.yml ps
# 所有 STATUS 应为 Up ... (healthy)

# 后端健康检查
curl http://127.0.0.1:8000/health
# → {"status":"ok"}

# 前端页面
curl -I http://127.0.0.1:3000
# → HTTP/1.1 200 OK

# 通过 Nginx 外部访问
```

## 本次部署架构

```
浏览器 → Nginx (443/80)
              ├── tomtranslate.com/api/*  → api 容器 (:8000)  → redis 容器 (:6379)
              │                                                → db 容器 (:5432)
              │                                                → libretranslate 容器 (:5000)
              └── tomtranslate.com/*      → web 容器 (:3000)
```

**关键配置映射**：
- `NEXT_PUBLIC_API_URL=https://tomtranslate.com`（同域路径模式，无需 api 子域名证书）
- `DEPLOYMENT_MODE=docker_host_nginx`（Nginx 在宿主机，其他服务在 Docker 内）
- 所有容器端口只绑定 `127.0.0.1`，外部流量必须经过 Nginx
