## 第 6 步：MVP 部署与发布

本步提供两条部署路线，根据你的需求选择：

| 路线 | 适合场景 | 运维复杂度 | 月成本 |
|------|---------|-----------|--------|
| **A. 托管平台（推荐 MVP）** | 快速验证、零运维 | ⭐ 极低 | ¥0 ~ ¥35 |
| **B. 自托管 VPS** | 完全可控、长期运营 | ⭐⭐⭐ 中等 | ¥50 ~ ¥150 |

---

### 6.1 路线 A：托管平台部署（推荐 MVP 阶段）

> 适合快速上线验证，不用管服务器、Docker、Nginx、HTTPS，全部由平台托管。

#### 6.1.1 后端 → Render

[Render](https://render.com) 支持直接从 GitHub 部署 Python 项目，免费套餐含 750 小时/月（足够 1 个服务 24h 运行）。

**步骤**：

1. 在 GitHub 推送代码（如果还没有远程仓库，先创建一个并推送）

2. 登录 [Render Dashboard](https://dashboard.render.com)，点击 **New → Web Service**

3. 连接 GitHub 仓库，配置如下：

   | 配置项 | 值 |
   |--------|---|
   | Name | `tom-translate-api` |
   | Root Directory | `packages/backend` |
   | Runtime | Python 3 |
   | Build Command | `pip install uv && uv pip install --system -e .` |
   | Start Command | `uvicorn app.main:app --host 0.0.0.0 --port $PORT` |

4. 在 Render 的 **Environment** 页面添加环境变量：

   ```
   DEEPSEEK_API_KEY=sk-你的真实key
   DEEPSEEK_BASE_URL=https://api.deepseek.com
   DEEPSEEK_MODEL=deepseek-chat
   GEMINI_API_KEY=你的真实key
   GEMINI_MODEL=gemini-2.5-flash
   REDIS_URL=（见下方 Upstash 配置）
   CORS_ORIGINS=https://你的前端域名.vercel.app,chrome-extension://*
   DEBUG=false
   API_SECRET_KEY=（生成一个 32 位随机字符串）
   RATE_LIMIT_PER_MINUTE=30
   ```

   > **注意**：`config.py` 中的 `env_file: "../../.env"` 在 Render 上不需要——Render 的环境变量会直接注入进程环境，`pydantic-settings` 会自动读取，优先级高于 `.env` 文件。

5. 部署完成后，Render 会生成一个地址如 `https://tom-translate-api.onrender.com`，访问 `/health` 验证：

   ```
   https://tom-translate-api.onrender.com/health
   → {"status": "ok"}
   ```

> **Render 免费套餐注意**：服务闲置 15 分钟后会休眠，首次请求需要 ~30s 冷启动。付费计划（$7/月）可保持常驻。

#### 6.1.2 Redis → Upstash

[Upstash](https://upstash.com) 提供 Serverless Redis，免费额度每天 10,000 请求，足够 MVP 验证。

**步骤**：

1. 注册 [Upstash Console](https://console.upstash.com)
2. 创建 Redis 数据库，区域选 **US-East-1**（或离你用户最近的区域）
3. 复制连接字符串（格式如 `rediss://default:xxx@xxx.upstash.io:6379`）
4. 填入 Render 的环境变量 `REDIS_URL` 中

#### 6.1.3 前端 → Vercel

1. 登录 [Vercel](https://vercel.com)，点击 **Add New → Project**
2. 导入 GitHub 仓库，配置：

   | 配置项 | 值 |
   |--------|---|
   | Root Directory | `packages/web` |
   | Framework Preset | Next.js（自动检测） |
   | Build Command | 留空（使用默认） |
   | Output Directory | 留空（使用默认） |

3. 在 **Environment Variables** 添加：

   ```
   NEXT_PUBLIC_API_URL=https://tom-translate-api.onrender.com
   ```

4. 点击 Deploy，完成后访问 Vercel 分配的域名即可使用网页翻译

> 后续每次 `git push` 到 main 分支，Vercel 会自动重新部署。

### 6.2 路线 B：自托管 VPS 部署

> 适合需要完全控制、长期运营、或国内用户为主的场景。

#### 6.2.1 服务器选型推荐

| 服务商 | 推荐配置 | 月费用 | 特点 |
|--------|---------|--------|------|
| **腾讯云轻量**（Hong Kong） | 2核2G，60G SSD | ¥34 ~ ¥50 | 免备案，Gemini API 可直连，国内访问延迟可接受 |
| **阿里云 ECS**（Hong Kong） | 2核2G，40G ESSD | ¥40 ~ ¥70 | 免备案，生态完善 |
| **Vultr / DigitalOcean** | 2核2G | $12/月 | 全球节点多，社区教程丰富 |
| **腾讯云轻量**（国内） | 2核4G，60G SSD | ¥50 ~ ¥80 | 国内访问快，但需域名备案（7~15 天） |

> **推荐 Hong Kong 节点**：免备案 + Gemini API 可直连 + 国内访问延迟 30~60ms，是 MVP 阶段最佳平衡。

#### 6.2.2 域名与 DNS

| 注册商 | 推荐 | 说明 |
|--------|------|------|
| **Cloudflare Registrar** | ⭐ 首选 | 成本价注册（无加价），自带 DNS + CDN + 免费 HTTPS |
| **NameSilo** | 备选 | 便宜，`.com` 约 $9/年 |
| **阿里云万网** | 国内备案用 | `.com` 约 ¥55/年，方便做备案 |

假设你购买了 `tomtranslate.com`，DNS 这样配置：

```
类型    名称              内容               代理
A       api               你的服务器IP       ✅（如果用 Cloudflare 代理）
CNAME   @                 cname.vercel-dns.com  ❌（Vercel 要求不代理）
```

- `api.tomtranslate.com` → 后端（VPS 上运行）
- `tomtranslate.com` → 前端（Vercel 托管）

#### 6.2.3 服务器初始化（SSH 连接后执行）

```bash
# 1. 更新系统
sudo apt update && sudo apt upgrade -y

# 2. 安装 Docker + Docker Compose
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# 重新登录使 docker 组生效
exit
# 重新 SSH 连接

# 3. 验证 Docker
docker --version
docker compose version

# 4. 创建项目目录
mkdir -p ~/tom-translate && cd ~/tom-translate

# 5. 克隆代码
git clone https://github.com/你的用户名/Tom_TranslateApp.git .
```

#### 6.2.4 后端 Dockerfile

创建 `packages/backend/Dockerfile`：

```dockerfile
FROM python:3.11-slim

WORKDIR /app

# 安装 uv（极速 Python 包管理器）
RUN pip install uv

# 先复制依赖文件，利用 Docker 层缓存（依赖不变时跳过安装）
COPY pyproject.toml uv.lock ./
RUN uv pip install --system -e .

# 复制应用代码
COPY app/ ./app/

EXPOSE 8000

# 使用 2 个 worker，生产环境不开 --reload
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "2"]
```

> **注意**：旧版手册中 `COPY ../../.env ./.env` 是错误的——Docker 不允许复制构建上下文之外的文件。正确做法是通过 `docker-compose.yml` 的 `env_file` 或环境变量注入配置。

#### 6.2.5 Nginx 配置

创建 `deploy/nginx/default.conf`：

```nginx
server {
    listen 80;
  server_name api.tomtranslate.com;

    location / {
    return 301 https://$host$request_uri;
    }
}

upstream tom_translate_api {
  server 127.0.0.1:8000;
  keepalive 32;
}

server {
    listen 443 ssl http2;
  server_name api.tomtranslate.com;

    ssl_certificate /etc/letsencrypt/live/api.tomtranslate.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.tomtranslate.com/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;

    location / {
    proxy_pass http://tom_translate_api;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
    }
}
```

> 这份配置是 **宿主机 Nginx** 用的，不放到 Docker 里跑。这样可以避免服务器上已有 Nginx/宝塔/Caddy 占用 443 时再和容器冲突。

#### 6.2.6 生产环境 Docker Compose

创建 `deploy/docker-compose.prod.yml`：

```yaml
services:
  api:
    build:
      context: ../packages/backend
      dockerfile: Dockerfile
    restart: always
    env_file:
      - ../.env
    # 覆盖 .env 中的相对路径配置，容器内不需要从文件读取（环境变量已注入）
    environment:
      - DEPLOYMENT_MODE=docker_host_nginx
      - REDIS_URL=redis://redis:6379/0
    ports:
      - "127.0.0.1:8000:8000"
    depends_on:
      - redis
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  redis:
    image: redis:7-alpine
    restart: always
    volumes:
      - redis_data:/data
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    networks:
      - app-network

volumes:
  redis_data:

networks:
  app-network:
    driver: bridge
```

> 生产 compose 默认只负责 **API + Redis**。80/443、证书签发与域名入口交给宿主机 Nginx 管理。

#### 6.2.7 生产环境变量

复制 `.env.production.example` 为 `.env`，修改以下关键项：

```bash
# === 必须修改 ===
DEPLOYMENT_MODE=docker_host_nginx
DEBUG=false
API_SECRET_KEY=用以下命令生成：python -c "import secrets; print(secrets.token_urlsafe(32))"
JWT_SECRET_KEY=用以下命令生成：python -c "import secrets; print(secrets.token_hex(32))"
CORS_ORIGINS=https://tomtranslate.com,https://www.tomtranslate.com,chrome-extension://*

# === 填入真实 API Keys ===
DEEPSEEK_API_KEY=sk-你的真实key
GEMINI_API_KEY=你的真实key

# === Redis（容器内网地址，由 docker-compose 自动解析） ===
REDIS_URL=redis://redis:6379/0

# === 不要写 localhost，必须是真实数据库主机名或容器服务名 ===
DATABASE_URL=postgresql+asyncpg://translate:你的密码@db-host:5432/translate_db
```

> 现在后端会校验 `DEPLOYMENT_MODE`。当模式不是 `local` 时，如果你把 `REDIS_URL`、`DATABASE_URL`、`LIBRE_TRANSLATE_URL` 或 `CORS_ORIGINS` 误写成 localhost，服务会在启动时直接报错，避免静默带病上线。

#### 6.2.8 首次部署流程

```bash
cd ~/tom-translate

# 1. 配置环境变量
cp .env.production.example .env
nano .env   # 填入真实配置（参考 6.2.7）

# 2. 启动后端与 Redis
docker compose -f deploy/docker-compose.prod.yml up -d --build

# 3. 验证容器内 API 已映射到宿主机 127.0.0.1:8000
curl http://127.0.0.1:8000/health

# 4. 把 deploy/nginx/default.conf 安装到宿主机 nginx 后重载
sudo cp deploy/nginx/default.conf /etc/nginx/conf.d/tom-translate-api.conf
sudo nginx -t
sudo systemctl reload nginx

# 5. 验证公网访问
curl https://api.tomtranslate.com/health
# → {"status": "ok"}
```

#### 6.2.9 日常运维命令

```bash
# 查看服务状态
docker compose -f deploy/docker-compose.prod.yml ps

# 查看后端日志（实时跟踪）
docker compose -f deploy/docker-compose.prod.yml logs -f api

# 更新代码并重新部署
git pull
```

### 6.3 前端部署（Vercel，两条路线通用）

无论后端走路线 A 还是 B，网页前端都推荐部署到 **Vercel**（Next.js 官方推荐，免费套餐够用）。

**步骤**：

1. 登录 [Vercel](https://vercel.com)，点击 **Add New → Project**
2. 导入 GitHub 仓库
3. 配置 Root Directory 为 `packages/web`
4. 在 **Environment Variables** 中添加：
   - `NEXT_PUBLIC_API_URL` = 你的后端地址（如 `https://api.tomtranslate.com` 或 `https://tom-translate-api.onrender.com`）
5. 点击 Deploy

**自定义域名（可选）**：
- 在 Vercel 后台 → Settings → Domains，添加 `tomtranslate.com`
- 按提示在 DNS 添加 CNAME 记录指向 `cname.vercel-dns.com`

> 后续每次 `git push` 到 main 分支，Vercel 会自动重新部署。

### 6.4 浏览器扩展发布

#### 6.4.1 构建扩展

```bash
cd packages/extension
pnpm build   # 产出在 .output/chrome-mv3/
pnpm zip     # 打包成 .zip（如果 WXT 支持；否则手动压缩 .output/chrome-mv3/ 目录）
```

#### 6.4.2 扩展中的 API 地址

扩展不需要在代码中硬编码后端地址——用户在 Popup 的 **设置 Tab** 中填写：
- **API 地址**：改为 `https://api.tomtranslate.com`（或 Render 分配的地址）
- **API Key**：填入与后端 `API_SECRET_KEY` 相同的值

当前代码已经改成：开发构建默认 `http://localhost:8000`，发布构建默认 `https://api.tomtranslate.com`。如果你的生产域名不同，再修改 `utils/settings.ts` 中的默认值即可。

#### 6.4.3 提交到 Chrome Web Store

1. 注册 [Chrome Web Store 开发者账号](https://chrome.google.com/webstore/devconsole)（$5 一次性，需 Google 账号 + 信用卡）
2. 上传 `.zip` 包
3. 填写商店信息：
   - 名称：`Tom Translate - 划词翻译 & 整页翻译`
   - 分类：`生产力工具`
   - 语言：`中文 (简体)` + `English`
   - 截图：至少一张 1280×800（展示划词翻译 + 整页翻译效果）
4. 提交审核（通常 1~3 个工作日）

#### 6.4.4 提交到 Edge Add-ons（可选，推荐）

共用同一套代码和 `.zip` 包，注册免费，审核通常更快：
- [Edge Add-ons 开发者中心](https://partner.microsoft.com/en-us/dashboard/microsoftedge/public/login)
- 上传同一个 `.zip`，填写商店信息后提交审核

### 6.5 部署后验证清单

上线后逐一验证以下功能，确保三端（后端、网页、扩展）协同正常：

| # | 验证项 | 预期结果 | 验证方法 |
|---|--------|---------|---------|
| 1 | 后端健康检查 | `{"status": "ok"}` | 浏览器访问 `https://api.xxx.com/health` |
| 2 | 生产环境文档已禁用 | 404 或空页面 | 浏览器访问 `https://api.xxx.com/docs` |
| 3 | CORS 正常 | 网页能正常翻译 | 在 Vercel 域名上点翻译，检查浏览器控制台无 CORS 报错 |
| 4 | API Key 认证生效 | 无 Key 时返回 401 | `curl -X POST https://api.xxx.com/api/v1/translate ...`（不带 X-API-Key 头） |
| 5 | Redis 缓存生效 | 相同请求秒回 | 同一段文字翻译两次，第二次响应时间 < 100ms |
| 6 | 网页版翻译正常 | 能翻译、切换引擎、切换语言 | 访问 Vercel 域名使用各功能 |
| 7 | 扩展划词翻译 | 选中文字 → 浮窗显示译文 | 安装扩展，修改设置中 API 地址为生产域名 |
| 8 | 扩展整页翻译 | 页面双语显示 + 进度条 | 打开英文页面，点击 Popup 的「翻译此页」 |
| 9 | 速率限制生效 | 高频请求返回 429 | 快速连续发送 31+ 请求 |
