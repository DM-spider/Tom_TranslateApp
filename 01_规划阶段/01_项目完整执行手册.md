## 第 1 步：项目初始化（第 1 天）

### 1.1 Monorepo 目录结构

项目采用 **Monorepo** 架构：将后端、网页前端、浏览器扩展、共享类型库放在同一个仓库中统一管理，便于代码共享和版本同步。

```
TranslateApp/
├── packages/
│   ├── backend/          # FastAPI 后端（Python）
│   ├── web/              # Next.js 网页前端
│   └── extension/        # 浏览器扩展（WXT）
├── shared/               # 前端/扩展共享的类型定义、工具函数
├── docs/                 # 文档
├── docker-compose.yml    # 本地开发基础设施（Redis）
├── pnpm-workspace.yaml   # pnpm Monorepo 配置
├── package.json          # 根目录统一脚本入口
├── .gitignore
├── .env.example          # 环境变量模板（提交到 Git）
└── README.md
```

### 1.2 根目录配置文件说明

**`pnpm-workspace.yaml`**：声明哪些子目录是 pnpm 工作区包。后端是 Python 项目，不参与此管理，只有 `web`、`extension`、`shared` 三个 JS/TS 子包被纳入。

**`package.json`（根目录）**：统一入口，定义以下开发脚本：
- `dev` — 同时启动后端 + 网页前端（通过 `concurrently` 实现）
- `dev:web` / `dev:ext` / `dev:backend` — 单独启动某个子项目
- `build:web` / `build:ext` — 分别构建前端页面和浏览器扩展

**`docker-compose.yml`**：用 Docker 运行 Redis，避免本地手动安装 Redis。PostgreSQL 服务已注释，等需要持久化用户数据时再启用。

**`.env.example`**：所有环境变量的模板文件，**提交到 Git**，作为配置说明文档，列出所有需要填写的密钥名称和示例值。实际使用时，复制为 `.env` 并填入真实密钥，`.env` 不提交到 Git。

**`shared/`**：TypeScript 类型定义库，存放前端网页和浏览器扩展共用的消息类型（翻译请求/响应格式、扩展内部通信协议等）。

### 1.3 初始化命令

```bash
# 安装前端依赖（在项目根目录执行）
pnpm install

# 复制环境变量模板，然后手动填入你的 API Keys（Windows 用 copy）
copy .env.example .env

# 启动 Redis（需要先安装 Docker Desktop）
docker compose up redis -d

# 验证 Redis 正常运行
docker compose ps
```

### 1.4 验证 Redis 正常运行（docker 安装不上）

```bash
# 检查 Redis 服务是否在运行
redis-cli ping  # 期望返回：PONG

# 进一步验证读写能力
redis-cli set test:hello "world"
redis-cli get test:hello
redis-cli del test:hello
# 期望依次返回：OK → "world" → (integer) 1

# 检查 Redis 监听端口
redis-cli info server

#确认输出中包含 tcp_port:6379，且你 .env 中配置的 REDIS_URL=redis://localhost:6379/0 与此一致。

```
如果以上都正常，Redis 没有问题。

## 第 2 步：后端开发（第 1 天 ~ 第 1 周结束）

### 2.1 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| Web 框架 | FastAPI | 高性能异步框架，自动生成 OpenAPI 文档 |
| 服务器 | Uvicorn | ASGI 服务器，支持热重载开发 |
| 配置管理 | pydantic-settings | 从 `.env` 文件自动读取并做类型校验 |
| HTTP 客户端 | httpx | 异步 HTTP，用于调用百度翻译等外部 API |
| AI SDK | openai | OpenAI 官方 SDK（同时支持 DeepSeek API） |
| 缓存 | redis[hiredis] | 异步 Redis 客户端，缓存翻译结果 |
| 包管理 | uv | 极速 Python 包管理器 |

### 2.2 后端目录结构

```
packages/backend/
├── app/
│   ├── __init__.py
│   ├── main.py                   # FastAPI 应用入口，注册路由和中间件
│   ├── config.py                 # 配置管理（从 .env 读取密钥）
│   ├── routers/
│   │   ├── __init__.py
│   │   └── translate.py          # 翻译 API 路由（/api/v1/translate）
│   └── services/
│       ├── __init__.py
│       ├── translator.py         # 翻译调度层（缓存 + 引擎选择）
│       ├── cache.py              # Redis 缓存逻辑
│       └── engines/
│           ├── __init__.py
│           ├── base.py           # 引擎抽象基类 + 数据模型定义
│           ├── deepseek.py       # DeepSeek 引擎实现
│           └── gemini.py         # Gemini 引擎实现
└── pyproject.toml                # 项目元数据 + 依赖声明

```

### 2.3 核心模块职责

**`config.py`（配置管理）**
使用 `pydantic-settings` 自动从 `.env` 文件读取配置项，进行类型校验，并通过 `lru_cache` 保证全局只初始化一次。

**`engines/base.py`（数据模型与基类）**
定义三个核心数据结构：
- `EngineType` 枚举：`deepseek`、`openai`、`baidu`
- `TranslateRequest`：翻译请求（文本列表、源语言、目标语言、引擎选择）
- `TranslateResult`：翻译结果（译文列表、检测到的语言、引擎名称、消耗的 token 数）

同时定义 `BaseEngine` 抽象基类，所有具体引擎都必须实现 `translate()` 方法，保证接口统一。

**`engines/deepseek.py`（核心 AI 翻译引擎）**
通过 OpenAI SDK 调用 DeepSeek API。核心策略：将多段文本用 `[SEP]` 分隔符合并为一个请求发送，返回时按 `[SEP]` 拆分，避免多次 API 调用，节省 token 消耗。

**`engines/baidu.py`（兜底引擎）**
直接调用百度翻译 HTTP API，无需 AI 能力，翻译速度快、成本低，适合 DeepSeek 额度耗尽时的降级处理。

**`cache.py`（Redis 缓存）**
对每段文本 + 源语言 + 目标语言 + 引擎的组合生成 SHA-256 哈希作为缓存键。命中缓存时直接返回，不消耗 API 额度。缓存有效期 7 天，支持批量查询。

**`translator.py`（调度层）**
翻译服务的主入口，执行以下逻辑：
1. 批量查询 Redis 缓存，找出未命中的文本
2. 仅将未命中的部分发给翻译引擎
3. 将新结果写入缓存
4. 合并缓存命中 + 新翻译结果，按原始顺序返回

**`routers/translate.py`（API 路由）**
对外暴露两个接口：
- `POST /api/v1/translate`：核心翻译接口
- `GET /api/v1/languages`：获取支持的语言列表

**`main.py`（应用入口）**
创建 FastAPI 实例，配置 CORS（允许网页前端和浏览器扩展跨域访问），注册路由，提供 `/health` 健康检查接口。

### 2.4 初始化与验证

```bash
# 进入后端目录并创建虚拟环境
cd packages/backend
uv venv
uv pip install -e ".[dev]"

*这一步相当于将后端的 python 单独隔离开了*

# 启动开发服务器（需先启动 Redis）

# 验证接口（另开一个终端）
curl -X POST http://localhost:8000/api/v1/translate \
  -H "Content-Type: application/json" \
  -d '{"texts": ["Hello, world!"], "target_lang": "zh-CN", "engine": "deepseek"}'
```

访问 http://localhost:8000/docs 查看自动生成的交互式 API 文档。

## 第 3 步：网页前端（第 2 周）

### 3.1 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| 框架 | Next.js 14（App Router） | React 全栈框架，支持 SSR/SSG |
| 样式 | Tailwind CSS | 原子化 CSS，快速构建 UI |
| 状态/请求 | @tanstack/react-query | 服务端状态管理，处理加载/错误状态 |
| 图标 | lucide-react | 轻量图标库 |

### 3.2 核心页面功能

只需要一个主页面，包含：
- **左栏**：输入文本框 + 源语言选择下拉菜单
- **右栏**：翻译结果展示 + 目标语言选择下拉菜单
- **顶部**：引擎切换（DeepSeek / 百度翻译）
- **功能按钮**：复制译文、清空输入、交换语言

### 3.3 API 通信层

位于 `src/lib/api.ts`，封装对后端的请求：
- 接收前端参数（驼峰命名），转换为后端格式（下划线命名）后发送请求
- 统一处理 HTTP 错误和网络异常
- 通过环境变量 `NEXT_PUBLIC_API_URL` 切换开发/生产后端地址

## 第 4 步：浏览器扩展（第 3-5 周）

### 4.1 技术选型

使用 **WXT**（Web Extension Tools）框架：
- 基于 Vite 构建，支持热重载开发
- 同时支持 Chrome / Firefox / Edge，无需维护多套配置
- 内置 TypeScript 支持

扩展权限：
- `activeTab`：读取当前标签页内容（整页翻译）
- `storage`：保存用户设置（默认语言、引擎偏好）
- `contextMenus`：右键菜单（右键翻译选中文字）
- `host_permissions: <all_urls>`：允许在任意网站注入内容脚本

### 4.2 开发顺序

**第一优先级：划词翻译**（1 周）

| 子任务 | 实现要点 |
|--------|---------|
| 监听选中事件 | `content.ts` 中监听 `mouseup`，用 `window.getSelection()` 获取选中文本 |
| 翻译图标 | 选中文本后在鼠标位置附近浮现一个小图标 |
| 翻译浮窗 | 点击图标后展开浮窗，显示原文 + 译文 |
| 通信机制 | Content Script → Background Service Worker → 后端 API（绕过跨域限制） |
| 设置存储 | `chrome.storage.local` 保存用户偏好（默认语言、默认引擎） |

**第二优先级：整页翻译**（2 周）

| 子任务 | 实现要点 |
|--------|---------|
| 触发入口 | Popup 中的「翻译此页」按钮 + 快捷键 `Alt+A` |
| DOM 解析 | 提取所有可见文本块，跳过 script/style/code 标签 |
| 智能分段 | 按 token 估算（1 中文字 ≈ 2 token，1 英文词 ≈ 1.5 token），每段 ≤ 1500 token |
| 并发翻译 | 同时发 3 个请求，用 `Promise.allSettled` 防止单个失败阻断整体 |
| 双语渲染 | 在原文下方插入译文节点，原文与译文同时可见 |
| 进度显示 | 页面右下角浮动进度条，显示翻译完成百分比 |
| 还原功能 | 一键移除所有插入的译文节点，恢复原始页面 |

### 4.3 扩展内部通信协议

Content Script（注入页面的脚本）不能直接调用后端 API（跨域限制），需要通过 Background Service Worker 中转。消息类型定义放在 `shared/` 目录，与前端网页共用。

**请求消息类型**：
- `TRANSLATE_SELECTION`：翻译选中文字
- `TRANSLATE_PAGE`：翻译整个页面
- `TRANSLATE_BATCH`：批量翻译文本段落
- `GET_SETTINGS` / `SAVE_SETTINGS`：读写用户设置

**响应消息类型**：
- `TRANSLATE_RESULT`：返回翻译结果
- `SETTINGS`：返回当前设置
- `ERROR`：返回错误信息

## 第 5 步：联调与打磨（第 5-6 周）

### 5.1 需要测试的场景矩阵

| 测试网站 | 关注点 |
|---------|--------|
| en.wikipedia.org | 长文本、表格、引用块 |
| github.com | 代码块需要跳过、Markdown 渲染 |
| news.ycombinator.com | 简单文本结构、嵌套评论 |
| twitter.com / x.com | 动态加载、虚拟列表 |
| medium.com | 富文本、图片与文字混排 |
| arxiv.org | 学术论文、数学公式需要保留 |
| amazon.com | 商品页面、多种布局混合 |

### 5.2 常见问题与解决方案

| 问题 | 解决方案 |
|------|---------|
| 动态加载内容（SPA）翻译不到 | 用 `MutationObserver` 监听 DOM 变化，新增节点自动翻译 |
| 译文导致排版变形 | 译文元素用 `display:block` 独立成行，不影响原文布局 |
| 翻译速度慢 | 懒翻译：先翻译视口内容，`IntersectionObserver` 监听滚动再翻译其余部分 |
| 大模型返回的分段数量不匹配 | 校验逻辑 + 回退：不匹配时逐条翻译 |
| 扩展与页面 CSS 冲突 | 使用 Shadow DOM 或极高特异性选择器隔离样式 |

## 第 6 步：MVP 部署与发布（第 7-8 周）

### 6.1 后端部署

**推荐方案**：阿里云 ECS（2核4G，约 ¥100/月）或腾讯云轻量应用服务器

部署方式：Docker Compose + Nginx 反向代理，三个服务组成完整的生产环境：
- `api`：FastAPI 后端容器（基于 `packages/backend/Dockerfile` 构建）
- `redis`：翻译缓存
- `nginx`：反向代理 + HTTPS 证书（Let's Encrypt 自动签发）

### 6.2 前端部署

推荐 **Vercel**（Next.js 官方推荐，免费套餐够用）：
- 连接 GitHub 仓库，自动检测 Next.js 项目，零配置部署
- 每次 push 到 main 分支自动触发部署
- 在 Vercel 后台配置环境变量 `NEXT_PUBLIC_API_URL` 指向后端地址

### 6.3 扩展发布

```bash
cd packages/extension
pnpm build   # 产出在 .output/chrome-mv3/
pnpm zip     # 打包成 .zip
```

1. 登录 [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole)
2. 上传 zip 包
3. 填写商店信息：名称、描述、截图（至少 1280×800）、分类选「生产力工具」
4. 提交审核（通常 1-3 个工作日）

同时提交到 **Edge Add-ons**（共用同一套代码，免费，审核更快）。
