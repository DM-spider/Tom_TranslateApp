## 第 1 步：项目初始化（第 1 天）

### 1.1 Monorepo 目录结构

项目采用 **Monorepo** 架构：将后端、网页前端、浏览器扩展、共享类型库放在同一个仓库中统一管理，便于代码共享和版本同步。

```
TranslateApp/
├── packages/
│   ├── backend/          # FastAPI 后端（Python）
│   ├── web/              # Next.js 网页前端
│   └── extension/        # 浏览器扩展（WXT）
├── shared/               # 前端/扩展共享的类型定义、工具函数
├── docs/                 # 文档
├── docker-compose.yml    # 本地开发基础设施（Redis）
├── pnpm-workspace.yaml   # pnpm Monorepo 配置
├── package.json          # 根目录统一脚本入口
├── .gitignore
├── .env.example          # 环境变量模板（提交到 Git）
└── README.md
```

### 1.2 根目录配置文件说明

**`pnpm-workspace.yaml`**：声明哪些子目录是 pnpm 工作区包。后端是 Python 项目，不参与此管理，只有 `web`、`extension`、`shared` 三个 JS/TS 子包被纳入。

**`package.json`（根目录）**：统一入口，定义以下开发脚本：
- `dev` — 同时启动后端 + 网页前端（通过 `concurrently` 实现）
- `dev:web` / `dev:ext` / `dev:backend` — 单独启动某个子项目
- `build:web` / `build:ext` — 分别构建前端页面和浏览器扩展

**`docker-compose.yml`**：用 Docker 运行 Redis，避免本地手动安装 Redis。PostgreSQL 服务已注释，等需要持久化用户数据时再启用。

**`.env.example`**：所有环境变量的模板文件，**提交到 Git**，作为配置说明文档，列出所有需要填写的密钥名称和示例值。实际使用时，复制为 `.env` 并填入真实密钥，`.env` 不提交到 Git。

**`shared/`**：TypeScript 类型定义库，存放前端网页和浏览器扩展共用的消息类型（翻译请求/响应格式、扩展内部通信协议等）。

### 1.3 初始化命令

```bash
# 安装前端依赖（在项目根目录执行）
pnpm install

# 复制环境变量模板，然后手动填入你的 API Keys（Windows 用 copy）
copy .env.example .env

# 启动 Redis（需要先安装 Docker Desktop）
docker compose up redis -d

# 验证 Redis 正常运行
docker compose ps
```

### 2.3 核心模块职责

**`config.py`（配置管理）**
使用 `pydantic-settings` 自动从 `.env` 文件读取配置项，进行类型校验，并通过 `lru_cache` 保证全局只初始化一次。

**`engines/base.py`（数据模型与基类）**
定义三个核心数据结构：
- `EngineType` 枚举：`deepseek`、`gemini`、`baidu`（百度暂未实现）
- `TranslateRequest`：翻译请求（文本列表、源语言、目标语言、引擎选择）
- `TranslateResult`：翻译结果（译文列表、检测到的语言、引擎名称、消耗的 token 数）

同时定义 `BaseEngine` 抽象基类，所有具体引擎都必须实现 `translate()` 方法，保证接口统一。请求验证器会自动过滤空白字符串，避免浪费 API 额度。

**`engines/deepseek.py`（核心 AI 翻译引擎）**
通过 OpenAI SDK 调用 DeepSeek API。核心策略：将多段文本用 `[SEP]` 分隔符合并为一个请求发送，返回时按 `[SEP]` 拆分，避免多次 API 调用，节省 token 消耗。安全措施：用户原文中的 `[SEP]` 会被转义为 `[[_SEP_ESCAPED_]]` 再发送，结果返回后还原，防止注入破坏分段。API 调用设有 30 秒超时保护。

**`engines/gemini.py`（备选 AI 翻译引擎）**
通过 Google genai SDK 调用 Gemini API。与 DeepSeek 引擎采用相同的 `[SEP]` 分段策略和防注入措施。使用 `client.aio.models.generate_content` 进行异步调用，同样设有 30 秒超时保护。

**`engines/baidu.py`（兜底引擎，暂未实现）**
直接调用百度翻译 HTTP API，无需 AI 能力，翻译速度快、成本低，适合 AI 引擎额度耗尽时的降级处理。当前为预留接口，选择百度引擎时会返回 400 错误提示。

**`cache.py`（Redis 缓存）**
对每段文本 + 源语言 + 目标语言 + 引擎的组合生成 SHA-256 哈希作为缓存键。命中缓存时直接返回，不消耗 API 额度。缓存有效期 7 天，支持批量查询。容错设计：所有 Redis 操作包裹 try/catch，Redis 宕机时静默降级（返回未命中），不阻断翻译流程，并设有连接超时 3s 和自动重试。

**`translator.py`（调度层）**
翻译服务的主入口，执行以下逻辑：
1. 批量查询 Redis 缓存，找出未命中的文本
2. 仅将未命中的部分发给翻译引擎
3. 将新结果写入缓存
4. 合并缓存命中 + 新翻译结果，按原始顺序返回

**`routers/translate.py`（API 路由）**
对外暴露两个接口：
- `POST /api/v1/translate`：核心翻译接口
- `GET /api/v1/languages`：获取支持的语言列表

异常处理已做安全脱敏：详细错误写入日志，返回给用户的是分类安全提示（认证失败/限流/超时/通用），不暴露内部配置或密钥。

**`main.py`（应用入口）**
创建 FastAPI 实例，注册路由，提供 `/health` 健康检查接口。包含以下安全中间件：
- **CORS**：允许网页前端和浏览器扩展跨域访问，方法限定为 `GET/POST/OPTIONS`，头部限定为 `Content-Type/X-API-Key`
- **速率限制**（`RateLimitMiddleware`）：基于 IP 的滑动窗口限流，仅对 `/api/` 路径生效，超限返回 429
- **API Key 认证**（`ApiKeyMiddleware`）：当 `.env` 配置了 `API_SECRET_KEY` 时，翻译接口需携带 `X-API-Key` 头，否则返回 401
- **文档访问控制**：`DEBUG=false` 时自动禁用 `/docs` 和 `/redoc`，防止生产环境暴露接口文档

### 3.4 API 通信层

位于 `src/lib/api.ts`，封装对后端的请求：
- 接收前端参数（驼峰命名），转换为后端格式（下划线命名）后发送请求
- 统一处理 HTTP 错误和网络异常，优先提取后端返回的 `detail` 错误信息
- 增加请求超时控制（AbortController），避免接口卡死导致前端长期 loading
- 增加非 JSON 响应兜底解析，后端异常页/网关错误时也能给出可读错误
- 通过环境变量 `NEXT_PUBLIC_API_URL` 切换开发/生产后端地址

### 3.8 稳定性优化

以下优化已在当前代码中落地，可作为后续开发基线：

1. **输入与交互防抖**
- 输入框使用 500ms 防抖触发翻译，减少高频请求
- 增加输入上限（5000 字符）与超长提示，避免超长文本导致体验劣化

2. **请求中的状态保护**
- 翻译进行中禁用关键交互：源/目标语言切换、交换语言、清空、复制、引擎切换
- 避免请求并发期间状态抖动和 UI 错乱

3. **错误可恢复能力**
- 新增 `src/app/error.tsx` 路由级错误边界
- 发生运行时异常时不再白屏，用户可点击“重试”或“刷新页面”恢复

4. **偏好持久化**
- 新增 `src/lib/storage.ts`，持久化 sourceLang / targetLang / engine
- 用户刷新页面后仍保留上次设置，减少重复配置成本

5. **React Query 重试策略优化**
- 4xx 业务错误不重试，网络类错误有限重试（最多 2 次）
- 提升错误反馈速度，减少无效重试

## 第 4 步：浏览器扩展（第 3-5 周）

### 4.1 技术选型

使用 **WXT**（Web Extension Tools）框架：
- 基于 Vite 构建，支持热重载开发
- 同时支持 Chrome / Firefox / Edge，无需维护多套配置
- 内置 TypeScript 支持

扩展权限：
- `activeTab`：读取当前标签页内容（整页翻译）
- `storage`：保存用户设置（默认语言、引擎偏好）
- `contextMenus`：右键菜单（右键翻译选中文字）
- `host_permissions: <all_urls>`：允许在任意网站注入内容脚本

│   └── popup/

| DOM 解析 | `dom-parser.ts`：提取所有可见文本块，跳过 script/style/code/pre/svg 等标签，过滤不可见元素 | ✅ |
| 智能分段 | `segmenter.ts`：按 token 估算（1 中文字 ≈ 2 token，1 英文词 ≈ 1.5 token），每段 ≤ 2000 token，不直接修改原始 TextBlock 对象 | ✅ |
| 并发翻译 | `page-translator.ts`：同时发 4 个请求，用 `Promise.allSettled` 防止单个失败阻断整体，支持 1 次重试 | ✅ |
| 双语渲染 | `renderer.ts`：在原文下方插入译文节点（浅蓝背景 + 蓝色左边框），支持「对照翻译」和「仅译文」两种显示模式 | ✅ |
| 进度显示 | `progress-ui.ts`：页面右下角浮动进度条（Shadow DOM 隔离），显示翻译完成百分比 | ✅ |
| 还原功能 | 一键移除所有 `[data-tom-translate]` 节点，恢复原始页面 | ✅ |

### 4.4 扩展内部通信协议

Content Script（注入页面的脚本）不能直接调用后端 API（跨域限制），需要通过 Background Service Worker 中转。消息类型定义放在 `shared/` 目录，与前端网页共用。

**请求消息类型**（Content Script → Background）：
- `TRANSLATE_SELECTION`：翻译选中文字（单条文本），payload: `{ text: string }`
- `TRANSLATE_BATCH`：批量翻译文本段落（整页翻译使用），payload: `{ texts: string[] }`
- `GET_SETTINGS` / `SAVE_SETTINGS`：读写用户设置

**页面控制消息类型**（Popup/Background → Content Script）：
- `TRANSLATE_PAGE`：触发整页翻译（由 Popup 按钮、快捷键 Alt+A 或右键菜单发出）
- `RESTORE_PAGE`：还原页面（移除所有译文）
- `GET_PAGE_STATE`：查询当前页面翻译状态
- `SHOW_TRANSLATION`：显示右键菜单翻译结果的浮窗

**响应消息类型**：
- `TRANSLATE_RESULT`：返回翻译结果
- `SETTINGS`：返回当前设置
- `PAGE_STATE`：返回页面翻译状态 `{ state, hasTranslations }`
- `ERROR`：返回错误信息

### 4.5 整页翻译技术细节

**翻译流程**：
1. 用户触发（Popup 按钮 / Alt+A / 右键菜单）→ 发送 `TRANSLATE_PAGE` 消息到 Content Script
2. 同语言检测：页面语言与目标语言相同时提示「无需翻译」，不浪费请求
3. `dom-parser.ts` 遍历 DOM 树，提取可见文本块，跳过 `<script>`、`<style>`、`<code>`、`<pre>`、`<svg>` 等标签
4. 视口优先排序：先翻译用户可见区域的文本块，再处理页面其余部分
5. `segmenter.ts` 按 token 估算将文本块分组，每组 ≤ 2000 token
6. `page-translator.ts` 以并发度 4 执行批量翻译，每个批次通过 `TRANSLATE_BATCH` 消息经 Background 调用后端 API
7. `renderer.ts` 在每个原文元素后插入译文节点，支持「对照翻译」和「仅译文」两种显示模式
8. `progress-ui.ts` 在右下角显示实时进度百分比
9. 翻译完成后显示"还原页面"按钮，点击后移除所有 `[data-tom-translate]` 节点

**容错设计**：
- 使用 `Promise.allSettled` 而非 `Promise.all`，单个批次失败不影响其他批次
- 每个批次支持 1 次重试（间隔 800ms）
- 失败批次计数显示在完成提示中
- 支持翻译过程中中止（`abort()`）

**动态内容支持**：
- 翻译完成后启动 `MutationObserver`，监听 DOM 新增节点
- 新增节点经 600ms 防抖后自动提取文本并翻译（适用于 SPA、无限滚动等场景）
- SPA 导航检测：每秒检查 URL 变化，切换页面时自动还原旧翻译

## 第 7 步：安全加固（已完成）

项目在功能开发基本完成后进行了一轮全面安全审计，修补了以下漏洞并落地到代码中：

### 7.1 后端安全

| 措施 | 说明 | 涉及文件 |
|------|------|---------|
| IP 速率限制 | 基于 IP 的滑动窗口限流，每分钟 30 次（可配置），超限返回 429 | `main.py` |
| API Key 认证 | `.env` 设置 `API_SECRET_KEY` 后，翻译接口需携带 `X-API-Key` 头 | `main.py`、`config.py` |
| 异常信息脱敏 | 错误详情只写日志，返回用户安全分类提示，不暴露内部配置 | `translate.py` |
| Debug 默认关闭 | `DEBUG` 默认 `False`，生产环境自动禁用 `/docs`、`/redoc` | `config.py`、`main.py` |
| 空文本过滤 | 请求验证器自动过滤空白字符串，避免浪费 API 额度 | `base.py` |
| CORS 收窄 | 方法限定 `GET/POST/OPTIONS`，头部限定 `Content-Type/X-API-Key` | `main.py` |
| [SEP] 防注入 | 用户原文中的 `[SEP]` 被转义后再发送，防止分段结果错位 | `deepseek.py`、`gemini.py` |
| AI 引擎超时 | DeepSeek / Gemini 调用 30 秒超时，防止网络异常时无限挂起 | `deepseek.py`、`gemini.py` |
| Redis 容错降级 | Redis 宕机时静默降级（跳过缓存），不阻断翻译流程 | `cache.py` |

### 7.2 扩展安全

| 措施 | 说明 | 涉及文件 |
|------|------|---------|
| API 请求超时 | 扩展端 fetch 增加 30 秒 AbortController 超时控制 | `utils/api.ts` |
| API Key 传递 | 扩展设置页支持填写 API Key，所有翻译请求自动携带 `X-API-Key` 头 | `settings.ts`、`api.ts`、`background.ts`、`App.tsx` |
| XSS 防护 | 进度条 UI 的 error message 经 `escapeHtml` 转义后再渲染 | `progress-ui.ts` |
| 无副作用分段 | `segmenter.ts` 不再直接修改输入的 TextBlock 对象 | `segmenter.ts` |
| 智能 Fallback | 批量翻译不匹配时仅补发缺失部分，不重发全部 | `background.ts` |

`.env.example` 新增以下配置项：
